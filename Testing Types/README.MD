## 🧪 Testing Types

Every time software is built or changed, something can break — or simply not work the way users expect. Testing Types is a structured set of checks applied throughout development to catch those problems before real users do. Some checks confirm that features work correctly; others confirm the product is fast, secure, and stable under real-world conditions. Together they answer two questions: *does it do what it should?* and *does it hold up when it counts?*

---

### Goals

**Business:**
- Reduces the cost of defects by catching them early — fixes before release are a fraction of the cost of fixes after
- Protects brand reputation by ensuring users receive a stable, secure, and performant product
- Provides documented evidence that acceptance criteria and compliance requirements are met
- Supports confident release decisions with clear pass/fail signal across all critical areas

**Development Team:**
- Gives developers fast, specific feedback on what broke, where, and why
- Prevents regressions — new code changes are verified not to break existing functionality
- Validates that both feature behavior and system quality (speed, security, reliability) meet defined thresholds
- Creates a shared, repeatable baseline that scales as the product and team grow

---

### Types

| Category | Focus | Examples |
|---|---|---|
| **Functional** | Does it behave correctly? | Unit, Integration, Regression, UAT, E2E |
| **Non-Functional** | Does it hold up in the real world? | Performance, Load, Security, Compatibility, Usability |
