## ⚙️ Functional Testing

### Why We Need It

**Business:**
- Confirms the product does what it was built to do — protects brand and user trust
- Catches requirement gaps before release, reducing costly post-launch fixes
- Provides documented proof that agreed acceptance criteria are met
- Reduces support and maintenance burden by shipping fewer defects

**Development Team:**
- Verifies that each feature behaves according to its specification
- Catches regressions when new code changes affect existing functionality
- Gives developers a clear pass/fail signal before merging or releasing
- Aligns QA and development on expected outcomes for every user action

---

### Aim
- Validate that all functional requirements are correctly implemented
- Cover positive, negative, and boundary scenarios for each feature
- Ensure the system handles valid and invalid inputs as specified
- Provide repeatable test evidence that the application meets its acceptance criteria

---

### Types

| Type | What It Verifies |
|---|---|
| **Unit Testing** | Individual functions or components work correctly in isolation |
| **Integration Testing** | Multiple modules interact and exchange data as expected |
| **System Testing** | The complete application meets all specified requirements end-to-end |
| **Smoke Testing** | Core critical paths are stable enough to proceed with further testing |
| **Sanity Testing** | A specific fix or change works without breaking related functionality |
| **Regression Testing** | New changes have not broken previously working features |
| **User Acceptance Testing (UAT)** | The product meets business requirements and is ready for release |
| **End-to-End Testing** | Full user workflows behave correctly across all integrated systems |

---

### Deliverables
- Functional test cases mapped to requirements or user stories
- Test execution results with pass/fail status per case
- Defect reports for any deviation from expected behavior
- Test coverage summary showing which requirements were verified
